# ADR 0007: Quest Assignment Mechanism

## Status
Accepted

## Context
Quests generated by the AI Engine currently exist in a vacuum. To integrate them into the gameplay loop, Players must be able to "Accept" quests. A quest needs to track which Player is executing it so that rewards can be correctly distributed by the GM upon completion, and so other players cannot claim the same unique quest simultaneously.

## Decision
We introduce an `AssigneeID` (string) to the `Quest` domain entity.
We introduce an `AcceptQuest(playerID string)` method on the `Quest` entity.
- A Quest can only be accepted if its state is `QuestStateAvailable`.
- Upon successful acceptance, the `AssigneeID` is set and the state automatically transitions to `QuestStateInProgress`.
- If the quest is already accepted, or is in an invalid state, it returns `ErrInvalidStateTransition` wrapped in our `Result[T]` type.

## Consequences
* **Positive:** Links the Player and Quest domains securely. Enforces mutual exclusion for quest acceptance.
* **Negative:** Modifies the core Quest state machine, slightly increasing complexity.
